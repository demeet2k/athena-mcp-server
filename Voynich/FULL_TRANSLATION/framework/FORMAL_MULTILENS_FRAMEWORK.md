# Formal Multilens Folio Framework

## Objective

Each folio is treated as a typed transformation program. The translation task is to express that program at four nested resolutions:

1. glyph and token
2. line
3. paragraph or block
4. whole folio

The same folio is then rendered through multiple codified languages while preserving one stable operational backbone.

## Base Objects

Let a folio `f` contain ordered lines `ell_1, ell_2, ..., ell_N`.

Each line `ell_j` has:

- an EVA string `E_j`
- a token decomposition `tau_j = (t_(j,1), ..., t_(j,m_j))`
- a direct operational reading `omega_j`
- a family of lens renderings `R_j^(lambda)`

The folio begins at an initial state `sigma_0` and ends at a terminal state `sigma_f`.

\[
\sigma_(j+1) = \Phi_j^(\mathrm{tot})(\sigma_j), \qquad j = 1, ..., N
\]

The dense folio file must show enough structure that every major claim can be replayed from `E_j` through `Phi_j^(\mathrm{tot})`.

## Line Grammar

The working VML grammar is:

\[
\text{token} = \text{operation} + \text{target} + \text{modifier} + \text{result state}
\]

At line scale:

\[
\Phi_j^(\mathrm{tot}) = O_(j,m_j) \circ O_(j,m_j-1) \circ \cdots \circ O_(j,1)
\]

where each `O_(j,k)` is chosen from the operator families in `registry/vml_operator_registry.md`.

## Paragraph And Folio Composition

If paragraph `P_r` contains lines `a_r, ..., b_r`, then:

\[
\sigma_(P_r) = \Phi_(b_r)^(\mathrm{tot}) \circ \cdots \circ \Phi_(a_r)^(\mathrm{tot})(\sigma_(P_(r-1)))
\]

with `sigma_(P_0) := sigma_0`.

The whole folio is:

\[
\sigma_f = \Phi_N^(\mathrm{tot}) \circ \cdots \circ \Phi_1^(\mathrm{tot})(\sigma_0)
\]

The final synthesis section must state what `sigma_f` means in direct operational language and in the simultaneous all-lens reading.

## Lens Transport Law

AQM is the primary formal source lens for the mathematical layer. Other mathematical lenses are generated by typed transport:

\[
\Phi_j^(\lambda) = T_\lambda^(-1) \circ \Phi_j^(\mathrm{AQM}) \circ T_\lambda
\]

where `T_lambda` is a domain transport from the AQM carrier to the carrier of lens `lambda`.

This law prevents the derived mathematical lenses from becoming free-floating metaphors. Each one must stay tethered to a typed source operator.

The lens registry uses the following default carriers:

- `AQM`: density operator `rho_n` on `H_tot`
- `CUT`: state vector `X_n = (kappa_n, varphi_n, ell_n, b_n)`
- `Liminal`: regime allocation triple `(p_n, lambda_n, f_n)`
- `Aetheric`: compression-expansion pair `(X_n, g_n)`
- `Chemistry`: composition vector `x_n^(chem)`
- `Physics`: field state `X_n^(phys)`
- `Quantum Physics`: density operator `rho_n^(qm)`
- `Wave Mechanics`: waveform `u_n`
- `Wave Math`: semigroup block `U_n`
- `Music Theory and Music Math`: pitch-tension vector `m_n`
- `Color and Light`: spectral density `I_n(lambda)`
- `Geometry`: path `gamma_n`
- `Number Theory`: count vector `a_n`
- `Compression`: compressed state `g_n`
- `Hacking Theory`: route `pi_n`

- `Story Writing`: plot state `S_n`
- `Hero's Journey`: arc state `H_n^(hero)`

The mythic carriers do not need the same quantitative rigor as the formal math carriers, but they still must be line-by-line, typed, and explicitly tied back to the operational backbone.

## Required Folio Anatomy

Every dense folio file must contain:

1. a full EVA block
2. a core VML machinery block
3. a direct line-by-line literal ledger
4. one subsection per canonical lens
5. direct operational meaning
6. mathematical extraction
7. mythic extraction
8. all-lens zero point
9. a formal mathematical overlay with imported kernels, state machine, line operators, safety invariants, transport laws, and a folio theorem

## Required Line Anatomy

Every line entry in the direct ledger must contain:

- line id
- EVA
- literal token chain
- operational English

Every formal-math line entry must contain:

- line id
- equation
- variable legend
- domain reading
- confidence note

Every mythic line entry must contain:

- line id
- frame or role
- movement
- reading
- confidence note

## Quality Gates

A folio is not complete unless all of the following are true:

- every visible line has an EVA entry
- every line appears in every canonical lens section
- every formal-math lens line entry contains at least one real equation
- the folio theorem composes the full ordered line sequence
- unresolved glyphs are still visible
- direct evidence and derived renderers remain clearly separated

## Section And Corpus Compilation

After a folio is complete:

1. append a compressed folio summary to `unified/VOYNICH_FULL_TRANSLATION.md`
2. append the folio's role to the relevant section synthesis file
3. add stations and transfers to the metro map
4. record completion in `manifests/CORPUS_BUILD_STATUS.md`

## Why This Framework Exists

The project is not just trying to paraphrase MS 408. It is trying to compile each folio into a stable, replayable, multilens object. The framework exists so the translation process remains cumulative, formal, and scalable as the corpus grows.
